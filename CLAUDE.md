## Project Overview

This is a "reverse Turing test" implementation where LLM judges try to identify their twin among other LLM candidates. The judge model interacts with candidates through messages and must guess which candidate is the same model as itself.

## Core Architecture

### Game Flow (turingtest.py)

The system uses a tool-based interaction pattern:

1. **Player (Judge)**: Uses `send_message` and `guess` tools via structured tool calls to interact with candidates
2. **Candidates**: Respond to judge's messages (can be LLMs or humans via Discord)
3. **Game**: Orchestrates turns, tracks probability estimates, saves trajectories to JSON

Key classes:
- `Player`: The judge LLM that uses tools to send messages and make guesses
- `Candidate`: A test subject (LLM) that responds to messages
- `Human`: A Discord-based player (responses come from actual humans)
- `Game`: Orchestrates the game loop and saves trajectories

### LLM Router (localrouter)

Uses the `localrouter` package for unified LLM API access. All LLM calls go through:
```python
from localrouter import get_response, ChatMessage, MessageRole, TextBlock
```

**Important**: Cache seeds are used throughout for response caching. Always pass `cache_seed` parameter to enable caching.

### Model Name Censoring

The `censor_model_names()` function in turingtest.py:258 removes model names from candidate responses to prevent judges from identifying themselves through text patterns. This is critical to maintain game integrity.

### Probability Tracking

After each round, the judge provides probability estimates for each candidate in a forked conversation (doesn't affect main history). Uses structured outputs with dynamically created Pydantic models (see Player.get_probability_estimate).

## Output Structure

### Game Trajectories

Games are saved to JSON files in output directories (e.g., `experiment1_results/`, `test_results/`):
- `game_TIMESTAMP_MODEL.json`: Full game data including metadata, trajectory, and probability estimates
- `index.json`: Generated by plot_results.py, lists all game files
- `analysis_results.json`: Generated by analyze_strategies.py, contains strategy and attribute analysis
- `plots/`: Probability evolution and confusion matrix visualizations

### Trajectory Format

Each round includes:
- `action`: "send_message" or "guess"
- `reasoning`: Judge's reasoning for the action
- `message`/`recipients`: For send_message actions
- `responses`: Candidate responses
- `probabilities`: Probability estimates for each candidate (0.0-1.0)
- `guess`/`correct`: For guess actions

## Key Implementation Details

### Tool Use Pattern

The judge model MUST use tools. If no tool call is found, the system prompts again. See Player.get_action() for retry logic.

### Structured Outputs

Used for:
- Probability estimates (dynamic Pydantic models in Player.get_probability_estimate)
- Strategy analysis (BatchAnalysisResult in analyze_strategies.py)

### Group Chat Mode

When enabled with `--group-chat`, all candidates see each other's responses, simulating a group conversation.

### Cache Seeds

Experiments use deterministic cache seeds (SHA256 hash of judge model + game number + candidates list) to ensure reproducibility across runs.

## Model Configuration

Edit the MODELS list in experiment1.py to test different model combinations. Supported prefixes:
- `anthropic/` or bare model names for Anthropic models
- `openai/` for OpenAI models
- `google/` for Google models

Model routing is handled by the localrouter package.

## Analysis Scripts

### plot_results.py

Generates:
- Probability evolution plots (one per judge model) showing how confidence changes over rounds
- Confusion matrices showing which models guessed which targets
- Statistics on win rates per model

### analyze_strategies.py

Uses an LLM (configured via ANALYZER_MODEL) to:
1. Identify strategies judges use to find their twin
2. Identify self-concept attributes models express
3. Process games in batches for consistency
4. Generate per-model breakdowns of strategy usage

## Important Notes

- The judge model's target is always itself (finding its twin)
- Candidate responses are censored to remove model name references
- Probability estimates are collected in conversation forks (don't affect main history)
- All games run asynchronously with asyncio for performance
- The viewer.html file requires a local web server to load JSON files due to CORS
